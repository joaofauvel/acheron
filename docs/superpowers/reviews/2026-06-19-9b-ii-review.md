# 9b-ii Adversarial Review

**Date:** 2026-06-19
**Reviewer:** subagent (adversarial)
**Scope:** Layer 9b-ii Redis async backend (commits `dd0c850..HEAD`)
**Verdict:** needs minor changes

## Summary

The async migration is mechanically correct: `redis.asyncio.Redis` is wired into both stores with proper `await`s, `aclose()` is idempotent in 7.4, the `connect()` instance method matches the spec, and the orchestrator calls it before registering local workers. Two real issues stand out: (1) a pre-existing idempotency bug in `Orchestrator.start()` where `_started = True` is set before `connect()` can fail, leaving the orchestrator permanently broken on Redis-down startup, and (2) a spec deviation that is actually correct but worth documenting. Most other concerns are cosmetic.

## Findings

### Important

**RED-9BII-1: `_started` flag is set before `connect()` runs, breaking retry semantics**
- **Location:** `src/acheron/shell/orchestrator.py:180-186`
- **Issue:** `start()` sets `self._started = True` *before* awaiting `self._registry.connect()`. If `connect()` raises (e.g., Redis unreachable at boot), the exception propagates but `_started` stays `True`. A subsequent `start()` call hits `if self._started: return` and exits immediately, so the orchestrator can never recover. This is a real correctness bug in the failure path, not a theoretical one — a supervisor loop or a `start()` call from a long-lived test harness would silently no-op. The same pattern existed in 9b-i (the `_started` flag was new in 86d82b3), but the new `connect()` call adds a network operation to the failure window, making the bug easier to hit.
- **Suggestion:** Either move `self._started = True` to *after* both `connect()` calls succeed, or wrap the post-`_started` block in `try/except` and reset `_started = False` in the except clause. The first option is simpler and race-free (asyncio is single-threaded).
- **Severity:** Important

**RED-9BII-2: `aclose()` on a Redis store with in-flight tasks tears down the pool mid-operation**
- **Location:** `src/acheron/shell/orchestrator.py:166-172` and `src/acheron/shell/stores/redis.py:276-278`
- **Issue:** `Orchestrator.close()` calls `await self._registry.close()` (which is `await self._redis.aclose()`) and `await self._job_store.close()`. The redis-py `aclose()` calls `disconnect()` on the entire pool, which forcibly closes in-flight connections. Any `submit_job` task currently running through `asyncio.create_task(self._execute(tracked))` will see its next `await self._job_store.put(tracked)` raise `ConnectionError`. The 9b-i docstring on `shutdown` already acknowledges that `_execute` tasks are not cancelled, but the same is not true for `close()` — the comment in `start()` is silent on this. This is the same shape as 9b-i's sync `close()`, but the async migration makes the timing of teardown more observable (the awaiting caller now actually waits).
- **Suggestion:** Add a one-line note in `close()` (or `shutdown()`) clarifying that callers must `shutdown()` and wait for in-flight jobs before `close()`. Optionally, `close()` could `await asyncio.gather(*self._tasks)` to drain pending work. A test that submits a job and then calls `close()` mid-flight would also lock this in.
- **Severity:** Important (subtle, but a real failure mode the new async code makes more likely to hit)

### Minor

**RED-9BII-3: Module-level comment explaining type ignores is verbose (5 lines, could be 2)**
- **Location:** `src/acheron/shell/stores/redis.py:3-7`
- **Issue:** The 5-line block-comment is justified (7+ `# type: ignore[misc]` instances need context), but it could be tighter. The current prose explains both mypy and basedpyright, the theoretical sync-usage branch, and the rationale. AGENTS.md asks for concise, timeless comments. A 2-line version conveys the same information.
- **Suggestion:** Compress to something like: ``"# redis.asyncio stubs type methods as ``Awaitable[T] | T`` (sync usage is theoretical); silence the misc."`` Keep the comment — it's needed, just shorter.
- **Severity:** Minor

**RED-9BII-4: Plan said "Drop all `# type: ignore[assignment]` comments" — implementation kept them as `[misc]`, which is correct but the spec is now wrong**
- **Location:** `src/acheron/shell/stores/redis.py:274, 304, 311, 338, 339, 365, 387` vs. `docs/superpowers/specs/2026-06-19-layer9b-ii-redis-async-design.md:159`
- **Issue:** The spec asserts "`redis.asyncio.Redis` types are correct" and tells the implementer to drop the type ignores. This is false — the redis-py async stubs type most methods as `Union[Awaitable[T], T]`, and the new `[misc]` ignores are required for both mypy strict and basedpyright standard. The implementation correctly kept the ignores (just changed the error code from `[assignment]` to `[misc]`), but the spec was not corrected. Future agents reading the spec would be confused.
- **Suggestion:** Update the spec at line 159 to say "Keep `# type: ignore[misc]` on each `await self._redis.<method>(...)` call — the redis-py async stubs type methods as `Awaitable[T] | T` and the `T` branch is unreachable in our async call sites." The new module-level comment already explains the rationale; the spec should match.
- **Severity:** Minor (spec drift, not a code bug)

**RED-9BII-5: `test_close_is_idempotent` only verifies "doesn't crash", not true idempotency**
- **Location:** `tests/shell/stores/test_redis_worker_store.py:136-143`
- **Issue:** The test calls `close()` twice and asserts no exception. This proves the second call doesn't raise, but not that the second call is a no-op. The class is named `TestCloseSemantics` and the docstring says "Verifies the pool is actually released" in the plan, but the actual implementation only tests crash-safety. A stronger test would spy on `aclose` (e.g., via `unittest.mock.patch.object`) and assert it was called once, or call `get()` between the two `close()` calls and assert the same outcome as a no-op second call.
- **Suggestion:** Either rename the test to `test_close_does_not_crash_when_called_twice` and adjust the class to `TestCloseRobustness`, or strengthen the test to verify the second call is a true no-op. The current name overpromises.
- **Severity:** Minor

**RED-9BII-6: `redis_url` fixture uses sync `asyncio.run()` instead of an async fixture**
- **Location:** `tests/shell/stores/conftest.py:19-34`
- **Issue:** The fixture is `@pytest.fixture` (sync) and calls `asyncio.run(_flush())` to drive an async FLUSHDB. This works but creates a fresh event loop per test, which is wasteful and inconsistent with the rest of the test suite (which uses `pytest_asyncio.fixture` and `async def` fixtures throughout). The plan's justification ("FLUSHDB is a one-shot prep, so sync is simpler") doesn't hold up — converting to `pytest_asyncio.fixture` would be ~3 lines and reuse the existing event loop.
- **Suggestion:** Convert to `pytest_asyncio.fixture` and `async def` + `yield` (matching the test fixtures in the same package). The conftest already imports `pytest` and `pytest_asyncio` is needed at the top.
- **Severity:** Minor

**RED-9BII-7: `from redis.exceptions import ...` is inside the test function, not at module top**
- **Location:** `tests/shell/stores/test_redis_worker_store.py:121` and `tests/shell/stores/test_redis_job_store.py:196`
- **Issue:** The import is inside the test method body, not at module top. This works but is inconsistent with the rest of the file (which imports `redis` at the top). The pattern is sometimes used to avoid import cost, but `redis.exceptions` is already loaded by `import redis`.
- **Suggestion:** Move both imports to the top of each file.
- **Severity:** Nit

**RED-9BII-8: `pipe.hset`, `pipe.sadd`, etc. inside `async with ... pipeline()` are not awaited — this is correct (plan was wrong) but worth confirming**
- **Location:** `src/acheron/shell/stores/redis.py:290-293, 297-300, 314-317, 348-351, 373-376, 390-393`
- **Issue:** The plan (Task 2, Steps 4-5) calls for `await pipe.hset(...)`, `await pipe.sadd(...)`, etc. inside the `async with` block. The implementation correctly omits the `await` on intermediate pipeline calls because they are sync buffer operations in redis-py (`pipeline_execute_command` returns the pipeline itself, not an awaitable). The plan was wrong; the implementation is right. Only `await pipe.execute()` is required. This is a positive spec deviation but could confuse future readers who compare the spec to the code.
- **Suggestion:** Add a one-line note to the spec (the plan, ideally not the design spec since the design is correct) clarifying that intermediate `pipe.<cmd>()` calls are sync. Alternatively, a 1-line comment near the first pipeline in `register()` would prevent re-introduction. Not strictly required.
- **Severity:** Minor (deviation is correct, just undocumented)

### Nit

**RED-9BII-9: `_SpyWorkerStore` / `_SpyJobStore` classes are defined inside the test function**
- **Location:** `tests/shell/test_orchestrator.py:107-117`
- **Issue:** The spy classes are local to the test. This is fine for one-off use, but the test pattern (subclass + override `connect()` to record + delegate) is a common shape that may want to be extracted to a shared helper if more such tests are added. For now, leave as is.
- **Suggestion:** No change. The spy classes are minimal and the test is the only consumer.
- **Severity:** Nit

**RED-9BII-10: `requires-python = ">=3.14"` and `pytest-asyncio==1.4` use the same `asyncio.run` loop in conftest — works in strict mode but a documented footgun**
- **Location:** `tests/shell/stores/conftest.py:33`
- **Issue:** Calling `asyncio.run()` from a sync fixture is allowed in pytest-asyncio 1.4 strict mode (the rule applies to test functions, not fixtures). However, if a future contributor tightens the config or moves the fixture into a `pytest_asyncio.fixture`, the `asyncio.run` call would interact badly with the running loop. This is not a bug today, but a comment near the call would help.
- **Suggestion:** One-line comment: "Sync fixture: we drive our own loop because pytest-asyncio only manages the test function's loop."
- **Severity:** Nit

**RED-9BII-11: `aclose()` is `async` but called from a sync `__init__` chain in some test paths — verify the fixture always awaits**
- **Location:** `tests/shell/stores/test_redis_worker_store.py:140` and `tests/shell/stores/test_redis_job_store.py:198`
- **Issue:** The `test_close_is_idempotent` test creates a store, calls `connect()`, `close()`, `close()` — all awaited. Good. But the user-visible pattern is "construct → connect → close" with explicit awaits. A future contributor could write `RedisWorkerStore(url).connect().close()` by accident (sync chain that produces a coroutine), and Python would silently produce a coroutine that's never awaited. The class is well-designed (all methods that need awaiting are `async def`), so this is more of a documentation note than a bug.
- **Suggestion:** No change. The class shape is correct; just noting that the contract is "always await" and the test demonstrates it.
- **Severity:** Nit

## What was checked and found clean

- **Spec compliance:** All required test plan items present (`test_unreachable_redis_raises_on_connect`, `test_connect_is_idempotent`, async `store` fixtures in both test files, `redis_url` uses `redis.asyncio`, spec doc update for 9b-ii at line 147 and 161).
- **Architecture match:** `__init__` does no I/O, `connect()` is an instance method (not classmethod), ABC has concrete no-op `connect()` default, `InMemory*Store` inherits the no-op without override, orchestrator awaits `connect()` before registering local workers. All match the design spec.
- **Pipeline correctness:** `async with self._redis.pipeline(...)` ensures cleanup on exception; `await pipe.execute()` is the only correct async point.
- **`exists(key)` semantics:** `if not await self._redis.exists(key)` returns False for missing keys (returns 0, which is falsy). Correct.
- **`aclose()` semantic in redis 7.4:** Inspecting the source confirms `aclose()` actually closes the pool (via `disconnect()`), not just marks it. With the default `auto_close_connection_pool=True`, this is correct.
- **`record_health_failure` → `unregister` path:** The `await self.unregister(worker_id)` inside `record_health_failure` is async-correct (was already correct in 9b-i; verified by reading the new code).
- **No surviving sync Redis calls in production code:** All call sites that previously used `self._redis.<method>()` are now awaited. `pipe.<method>()` calls are correctly sync-buffer (intentional, see RED-9BII-8).
- **Test structure:** Spy classes are necessary (the test needs to verify `connect()` was awaited on both stores; no simpler way exists without breaking the `WorkerStore` ABC contract).
