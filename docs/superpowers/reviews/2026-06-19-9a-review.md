# 9a Adversarial Review

**Date:** 2026-06-19
**Reviewer:** subagent (adversarial)
**Scope:** Layer 9a streaming executor (commits `5644116..HEAD`)
**Verdict:** needs changes

## Summary

The streaming executor's core architecture is sound: per-stage `asyncio.Queue`, outer `TaskGroup` cancellation, sentinel-driven drain, and cache-sourced `PlanResult.outputs` all work. The exception-placement test, the default-strategy change, and the async cache migration are clean. The implementation, however, has two real correctness defects (`_collect_outputs` does not catch `CacheCorruptedError`; `total_cost` is hardcoded to `0.0`), plus several spec-compliance deviations around error message format and sentinel-violation handling. Tests cover the surface of the spec but several are tautological (the cache-content test) or don't actually exercise the path they claim (the cancellation test).

## Findings

### Important

**RED-9A-1: `_collect_outputs` does not catch `CacheCorruptedError` from a partial save**
- **Location:** `src/acheron/shell/executors/streaming.py:115-126`
- **Issue:** `StepCache.load_outputs` raises `CacheCorruptedError` for both an `OSError` on read *and* an invalid JSON parse (cache.py:99-105). A stage that is cancelled mid-`save_outputs` (a real possibility inside a `TaskGroup` whose sibling raised) leaves a partial manifest on disk. The next call to `_collect_outputs` then raises `CacheCorruptedError` and crashes the executor with an unhandled exception, instead of producing a `failed` `PlanResult`. The current `except` only catches `CacheMissError` and `OSError`, missing the partial-write case entirely. This is the same kind of failure the spec says the cache layer is supposed to be resilient to ("`CacheCorruptedError` or unexpected `OSError` from `save_outputs` → wrapped in `PipelineError`").
- **Suggestion:** Change the catch to `except (CacheMissError, CacheCorruptedError): continue` (and keep `except OSError: logger.exception(...); continue`). Add a regression test that writes a partial/invalid manifest to the cache step dir, runs a plan, and asserts `result.status == "failed"` (not a raised exception).
- **Severity:** Important

**RED-9A-2: `total_cost=0.0` hardcoded in `_build_result` and `_empty_result`**
- **Location:** `src/acheron/shell/executors/streaming.py:144, 154, 198`
- **Issue:** Every other executor accumulates `result.metrics.cost_estimate or 0.0` per completed step (`async_executor.py:56, 61`, `sequential.py:46`, `batch_async.py:60, 65`). The streaming executor hardcodes `total_cost=0.0` in all three `PlanResult` construction sites. The cost information from the per-step `JobResult.metrics` is silently discarded, and the `PlanResult` returned for an actual run is materially different from what `BatchAsyncExecutor` (still in the codebase, registered in the factory) would produce for the same plan. Any cost-tracking downstream of `PlanResult.total_cost` (e.g., the API response in `JobResponse.total_cost`, dashboards, billing) will report zero for every streaming run.
- **Suggestion:** Track `total_cost` in the success path of `_build_result`. Concretely: have `_stage` return the `JobResult` (or have `_run_pipeline` re-read the just-written manifest and sum `cost_estimate` from each), then pass the accumulated `total_cost` into `_build_result`. Add a test that constructs a `JobMetrics(cost_estimate=1.23)` and asserts `result.total_cost == 1.23` (and `2.46` for two steps).
- **Severity:** Important

**RED-9A-3: `completed_steps=0` on failure discards partial-success information**
- **Location:** `src/acheron/shell/executors/streaming.py:151`
- **Issue:** When a stage fails partway through a run, the executor collects the outputs of every stage that *did* write a manifest (`_collect_outputs` reads the whole cache), but `completed_steps=0` is hardcoded on the failure path. The other three executors report the actual count of successful steps. This contradicts the dataclass semantics ("`completed_steps: int`") and is observable by clients: the `JobResponse.completed_steps` field will be `0` even when several stages' outputs are present in `result.outputs`. The spec acknowledges partial state "never arises" in linear pipelines, but it is reachable as soon as a stage succeeds and a later one fails — which is exactly the case `_collect_outputs` is built to handle.
- **Suggestion:** Count successful steps during `_collect_outputs` (e.g., count steps where `load_outputs` returned without raising `CacheMissError`) and pass that count to `_build_result`. On failure, set `completed_steps=that_count` instead of `0`. Add a test where stages 0 and 1 succeed and stage 2 raises, asserting `result.completed_steps == 2`.
- **Severity:** Important

**RED-9A-4: Unexpected-exception message format does not match the spec**
- **Location:** `src/acheron/shell/executors/streaming.py:108-110`
- **Issue:** The spec's exception-wrapping table says: `Unexpected Exception in a stage | PipelineError("unexpected failure in stage {id}: {type}") from exc`. The implementation does not wrap unexpected exceptions in the stage; it lets them propagate to `_extract_error`, where they become `PipelineError("streaming failure: {inner}")`. The stage id and exception type are lost from the message. The wrapping is functionally a `PipelineError` (so the test passes), but the diagnostic value is materially lower — operators reading `PlanResult.errors` cannot tell which stage failed or what type of exception it was. The stage-level try/except that exists for `TimeoutError` and `save_outputs` is the natural place to add a third branch for the unexpected case.
- **Suggestion:** Add a third `try/except Exception` around the handler dispatch in `_stage` that raises `PipelineError(f"unexpected failure in stage {step.step_id}: {type(exc).__name__}") from exc`. Update the `TestUnexpectedException` assertion to check for the new format.
- **Severity:** Important (spec deviation, real diagnostic loss)

### Minor

**RED-9A-5: Dead code: seed `_END` in `queues[0]` is never consumed**
- **Location:** `src/acheron/shell/executors/streaming.py:57` (paired with `queues[i] if i > 0 else None` at line 77)
- **Issue:** The plan called for seeding `queues[0]` with `_END` so the first stage could "read the seed sentinel and dispatch the first step." The refactor (e89078f) made the first stage skip the upstream read entirely (`upstream=None`), so the seed is written but never read. This is harmless — the comment is now misleading. It also means `queues[0]` is created, seeded, and never used, which is wasted work and reader confusion.
- **Suggestion:** Either remove the `await queues[0].put(_END)` and the comment, or revert the first-stage upstream to `queues[0]` so the seed has a purpose. The former is consistent with the refactor's direction.
- **Severity:** Minor

**RED-9A-6: Sentinel protocol violation is silently swallowed**
- **Location:** `src/acheron/shell/executors/streaming.py:170-171`
- **Issue:** The spec's exception table lists `Sentinel protocol violation (e.g., None arrives mid-stream) → PipelineError("sentinel received before queue drained") from exc`. The implementation's `_stage` just `return`s when it sees `_END`, with no distinction between "expected drain after upstream finished" and "premature termination mid-stream." This is acceptable for the current linear topology (where the second case cannot arise), but the spec explicitly calls for the check. If a future branchy topology reuses this executor, the missing check will silently drop data.
- **Suggestion:** Track whether the stage has received an expected number of upstream results, and raise `PipelineError` if `_END` arrives early. For now, a `# TODO(linear-only):` comment near the check is sufficient.
- **Severity:** Minor (linear-only topology makes the gap latent; flag for the next branchy layer)

**RED-9A-7: `_extract_error` does not filter `CancelledError` from `eg.exceptions`**
- **Location:** `src/acheron/shell/executors/streaming.py:103-113`
- **Issue:** When a stage fails with a non-`AcheronError` exception (e.g., `RuntimeError`), the `BaseExceptionGroup` contains both the original exception and `CancelledError`s from cancelled siblings. The implementation picks `eg.exceptions[0]` to wrap in a `PipelineError`. Python does not guarantee a stable order in the group; if a `CancelledError` ends up at index 0, the wrapped error becomes `PipelineError("streaming failure: ")` — a misleading message with no actual cause text. The filter for `AcheronError` at the top of the method does the right thing, but the second branch's `eg.exceptions[0]` access is order-dependent.
- **Suggestion:** Filter out `BaseException` (specifically `CancelledError`) when picking the inner exception: `inner = next((e for e in eg.exceptions if not isinstance(e, BaseException) or not isinstance(e, CancelledError)), eg.exceptions[0])`. Or simply guard with `if not isinstance(inner, BaseException)`. The current tests don't exercise the bad-order case, so this is latent.
- **Severity:** Minor

**RED-9A-8: `test_outputs_match_cache_contents` is tautological**
- **Location:** `tests/shell/test_streaming_executor.py:259-275`
- **Issue:** The test verifies `result.outputs` has 4 items, then for each step verifies that the cache has the same number of items. But the executor *writes* the handler's outputs to the cache and *reads* them back into `result.outputs` — so the test is verifying the executor's own write-then-read round trip, not that the cache is the source. The spec's test-plan note said "verifiable by deleting the in-memory outputs and showing the cache is the source"; the current test does not do this. If `_collect_outputs` were broken (e.g., reading from a stale in-memory list), the test would still pass as long as the stage wrote the right things to the cache and the cache was readable.
- **Suggestion:** Either (a) add a `monkeypatch` that intercepts `step_cache.load_outputs` and returns a *different* `tuple[OutputFile, ...]` from what the handler returned, then assert `result.outputs == patched_outputs`; or (b) directly assert that `_collect_outputs` is called (e.g., by injecting a spy on the cache load path). Option (a) is the more direct lock-in.
- **Severity:** Minor

**RED-9A-9: `test_middle_failure_cancels_upstream_and_downstream` does not actually verify cancellation**
- **Location:** `tests/shell/test_streaming_executor.py:202-230`
- **Issue:** The test asserts `not package_manifest.exists()`. This is true if the package stage was cancelled mid-handler, but it is *also* true if the stage was never dispatched (which happens here: the package stage reads `_END` from `queues[2]` after chunk's `finally` put it there, and returns via the `if upstream_value is _END: return` early-exit before the handler is ever called). So the test passes for the wrong reason: the sentinel-drain path, not the TaskGroup cancellation path. The test is misnamed and misdescribed; a `class TestTaskGroupCancellation` member should verify cancellation, but the cancellation isn't observable in this configuration.
- **Suggestion:** Either rename the test to `test_middle_failure_drains_downstream_via_sentinel` (and move it to `TestSentinelDrain`), or make the package stage block on something other than the queue (e.g., a long `asyncio.sleep` triggered by a `step.payload` flag) and assert that the sleep was interrupted with `CancelledError`. The latter is what the spec test plan called for.
- **Severity:** Minor

**RED-9A-10: Orchestrator import for `StepCache` is placed awkwardly**
- **Location:** `src/acheron/shell/orchestrator.py:27-34`
- **Issue:** The file has a `if TYPE_CHECKING:` block at lines 27-32 with `from acheron.shell.cache import PlanCache`, then at line 34 an unconditional `from acheron.shell.cache import StepCache` sits below it. The split makes sense (`PlanCache` is a type hint only, `StepCache` is constructed at runtime), but the placement — a single import *below* the `TYPE_CHECKING` block, separated from the rest of the import group — looks like an oversight rather than a deliberate split. Other Acheron files group runtime imports together at the top.
- **Suggestion:** Move the `from acheron.shell.cache import StepCache` to the top import group (alongside `from acheron.core.errors import AcheronError`, etc.), and keep `PlanCache` inside the `TYPE_CHECKING` block. Two imports, one for runtime, one for typing, both at the conventional position.
- **Severity:** Minor (style; pattern works as written)

**RED-9A-11: `_END: None = None` is an unusual sentinel for a typed queue**
- **Location:** `src/acheron/shell/executors/streaming.py:27`
- **Issue:** The sentinel is a module-level `None` value, and the queue type is `asyncio.Queue[JobResult | None]`. This is legal because `JobResult` is never `None`, but the type signature does not make the sentinel role explicit — a reader has to know the convention. The `is _END` checks throughout `_stage` work, but a more typed approach (`_EndSentinel` class or a `Final` constant) would let the type system help.
- **Suggestion:** Acceptable as-is, but consider `_END: Final = object()` and a separate `JobResult | _EndSentinel` queue type if a typed sentinel class is desired. The current form is consistent with the `StepHandler` style elsewhere in the project, so this is mostly a nit.
- **Severity:** Minor

**RED-9A-12: The spec's expected-exception-wrapping spec for `Unexpected Exception` deviates from implementation**
- **Location:** `src/acheron/shell/executors/streaming.py:172-179`
- **Issue:** The stage-level `try/except TimeoutError` is the natural place to also catch a bare `Exception` from `self._handler(...)` and wrap it. The implementation does not, so the wrapping is delegated to `_extract_error`, which loses the stage id and type name. See RED-9A-4 for the message-format issue. The two findings are related but separable.
- **Suggestion:** Add a third `except Exception` branch in `_stage` that wraps as `PipelineError("unexpected failure in stage {step.step_id}: {type(exc).__name__}") from exc`. This puts the wrapping in the right place and makes the spec table literally accurate.
- **Severity:** Minor (already counted under RED-9A-4; listed here for traceability of the spec deviation)

### Nit

**RED-9A-13: `except X, Y, Z:` (Python 2-style) is unusual in Python 3.14**
- **Location:** `src/acheron/shell/cache.py:114`
- **Issue:** The line `except CacheMissError, CacheCorruptedError, OSError:` is Python 2 syntax for tuple exceptions. Python 3.14's `ast` parses it as a tuple (verified by running `ast.parse`), so the code is valid, but readers familiar with Python 3 expect `except (X, Y, Z):` with explicit parens. Ruff is currently silent (config doesn't flag this), but it remains surprising.
- **Suggestion:** Add the parentheses: `except (CacheMissError, CacheCorruptedError, OSError):`. Same behavior, easier to read.
- **Severity:** Nit

**RED-9A-14: `step_cache._data_dir` private access in test**
- **Location:** `tests/shell/test_streaming_executor.py:229` (and `:194` uses `_data_dir` in `_real_output` helper)
- **Issue:** The test reaches into `step_cache._data_dir` to assert that the package manifest does not exist. This is suppressed with `# noqa: SLF001` (good), but a public accessor would be cleaner. `StepCache` already has a `data_dir` property on `PlanCache` (cache.py:34-37); the same property on `StepCache` would let the test write `step_cache.data_dir / ...`.
- **Suggestion:** Add a `data_dir` property to `StepCache` mirroring `PlanCache.data_dir`, and use it in the test. One-liner change.
- **Severity:** Nit

**RED-9A-15: `last_error.__cause__ = inner` could be `raise ... from inner` if restructured**
- **Location:** `src/acheron/shell/executors/streaming.py:110-111`
- **Issue:** The pattern `err = PipelineError(...); err.__cause__ = inner; return err` is the documented way to chain a cause without using `raise` (since the function wants to return, not raise). The implementation is correct. This is a nit only because AGENTS.md asks for clean exception chaining; the current code achieves the same effect without `raise`, which is the right choice here.
- **Suggestion:** No change. The current pattern is correct for a return-instead-of-raise context.
- **Severity:** Nit

**RED-9A-16: `isinstance(first, AcheronError)` check in `_consume_final_queue` is dead code**
- **Location:** `src/acheron/shell/executors/streaming.py:90-93`
- **Issue:** No stage ever puts an `AcheronError` into a queue — stages only put `JobResult` or `_END`. The check is defensive but unreachable. If a stage ever did put an error (which would require an API change), the check would matter; as written, it is a guard against a hypothetical.
- **Suggestion:** Either remove the check (and the unused `AcheronError` import in this method's scope) or add a comment explaining the defensive intent. The check costs nothing at runtime but adds a reader detour.
- **Severity:** Nit

**RED-9A-17: `failed` `PlanResult` carries `completed_steps=0` even when `len(outputs) > 0`**
- **Location:** `src/acheron/shell/executors/streaming.py:148-156`
- **Issue:** This is the same root issue as RED-9A-3, but worth noting as a separate nit that the `errors` field is a `tuple[str, ...]` built from a single `str(last_error)` rather than a structured `tuple[AcheronError, ...]`. The string-form is what other executors do, so this is consistent, but it loses the error type (you can't tell from `result.errors` whether it was a `WorkerError` or a `PipelineError`).
- **Suggestion:** Acceptable as-is; consistent with the other executors. Listed here for completeness.
- **Severity:** Nit

## What was checked and found clean

- **Spec compliance — test plan:** All 8 spec test cases are present (`test_three_step_plan_completes`, `test_slow_handler_raises_worker_error`, `test_worker_unavailable_propagates`, `test_unhandled_exception_wrapped_as_pipeline_error`, `test_save_outputs_failure_wrapped_as_pipeline_error`, `test_middle_failure_cancels_upstream_and_downstream`, `test_sentinel_propagates_downstream`, `test_outputs_match_cache_contents`). Several are weak (see Minor), but none are missing.
- **Sentinel protocol:** The `_END` propagation works as designed — the `finally: await downstream.put(_END)` in `_stage` ensures every stage always writes a sentinel on its downstream queue. The `_consume_final_queue` drain task exits cleanly when it sees the sentinel. TaskGroup cancellation is handled correctly for the happy path and the failure path.
- **Per-step timeout:** `asyncio.wait_for(self._handler(...), timeout=self._step_timeout)` is the only dispatch point, so every handler call is wrapped. `TimeoutError` → `WorkerError` wrapping is correct.
- **`PipelineError` placement:** `PipelineError` is a subclass of `AcheronError` and not of `WorkerError`, as required by the spec and verified by the placement test in `test_errors.py`.
- **StepCache async migration:** `save_outputs`/`load_outputs`/`step_has_valid_cache` are all properly `async def`; `aiofiles` is used for the manifest I/O; `asyncio.to_thread` is used for the blocking checksum. `tests/shell/test_cache.py` was correctly converted to `async def` with `pytest_asyncio.fixture`.
- **Type hygiene:** `mypy` strict passes on all 41 source files; `basedpyright` standard passes on `streaming.py`. The `JobResult | None` queue type is correct (None is the sentinel). `topological_order` is reused; no reinvented dependency walker.
- **Default-strategy change:** `api/schemas.py:15` and `api_client.py:27` both default to `"streaming"`. The CLI's `--executor` default is intentionally left at `"batch_async"` per the spec, which is consistent.
- **Factory wiring:** `create_executor(ExecutorStrategy.STREAMING, ...)` requires `step_cache` (raises `ValueError` otherwise), and the orchestrator constructs `self._step_cache = StepCache(cache.data_dir)` and passes it. The orchestrator also correctly catches `AcheronError` and `Exception` from `executor.run`.
- **`_run_pipeline` → `BaseExceptionGroup` → `_extract_error`:** The exception-group unwrapping pattern correctly prefers the first `AcheronError` and falls back to wrapping the first non-AcheronError in a `PipelineError`. The `__cause__` chaining is set correctly.
- **No new sync `StepCache` callers left:** The only callers of `StepCache` are the executor (uses `await`), the cache tests (converted to `async def`), and the orchestrator (constructs and passes through).
- **Conventional Commits:** All 9a commit messages follow the format (`feat(...)`, `test(...)`, `chore(deps):`).
- **TDD discipline:** Each test was added before the implementation that made it pass; commit history shows the failing-test-then-pass progression. (Verified by reading the plan and commit log.)
- **Integration with existing tests:** The integration test suite does not hardcode `"batch_async"` (the only hit in `grep` was the `test_submit_invalid_executor_strategy` test, which is correct). The CLI test still works because the CLI default is unchanged.
- **`is_last` removal:** The plan included `is_last` in the `_stage` signature; the implementation dropped it (the "last stage" behavior is handled by `_consume_final_queue`). The simplification is correct and matches the refactor in e89078f.
- **`PipelineError` for `save_outputs`:** The `try/except Exception` in `_stage` correctly wraps any save failure as `PipelineError("save_outputs failed for step {step.step_id}")`, with `from exc` for the cause chain. The test for this is correct.
